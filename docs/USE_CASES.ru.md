# Сценарии использования — Sanskrit Karaoke

Проект обслуживает **две аудитории**, и этот документ построен вокруг них:

- **Мейнтейнеры** — те, кто разрабатывает приложение и эксплуатирует конвейер
  производства видео (разработчики, продюсеры дропов, менеджер прав, куратор библиотеки).
- **Конечные пользователи** — те, ради кого всё делается: зрители, приходящие из
  соцсетей по воронке, и студенты, системно заучивающие шлоки.

Технически в проекте два слоя:

1. **Браузерное приложение** (старый слой) — `index.html` для авторской работы и
   `student.html` / `catalogue.html` / `progress.html` для студентов.
2. **Конвейер пакетного производства видео** (слой 2026 года) — из аудиозаписи целой
   главы получаются готовые к публикации вертикальные карaoке-видео (`feed_v1`) с
   воронкой к платному курсу.

# Часть I. Мейнтейнеры

## 1. Разработчик — локальная настройка и архитектура
**Цель:** запустить проект локально и понять, где что лежит.

### Рабочий процесс:
1. **Запуск:** сборка и пакетный менеджер не нужны — `python -m http.server 8000`,
   открыть `http://localhost:8000`.
2. **Проверка синтаксиса:** `node --check src/scripts/app.js`.
3. **Архитектура ядра:** логика вынесена из монолита `app.js` в чистые ES-модули
   `src/core/*` (ADR-0001): `translit`, `layout`, `svg`, `compose`, `karaoke-frame`,
   `timing` и **`feed`** (вертикальный шаблон `feed_v1`). Модули без DOM/глобалей;
   `app.js` их оборачивает.
4. **Безголовый рендер:** `render.html` загружается Puppeteer-ом и вызывает
   `window.renderVerse()` — это поверхность для пакетного рендера видео (без UI).
5. **Валидация и индекс:** `python tools/validate_library.py` (схема + проверки
   готовности к публикации), `python tools/build_index.py` (пересборка индекса каталога).
6. **Конвенции:** журнал сессий — `.ai_state.md`; архитектурные решения — `docs/adr/*`;
   все файлы UTF-8 **без BOM**.

---

## 2. Продюсер дропа — «глава → видео → посты» одной командой
**Цель:** превратить пакетную аудиозапись главы в набор готовых к публикации
карaoке-видео с подписями, хэштегами и ссылками воронки. Это ядро нового слоя.

### Рабочий процесс:
1. **Аудио:** Уша Санка записывает целую главу за одну сессию (батч-запись). Файлы
   называются по `id` шлоки — `bhg_2_47.m4a`, `bhg_2_48.m4a` и т. д. — и кладутся в
   одну папку.
2. **Зависимости рендера:** один раз — `npm install --prefix tools` (Puppeteer).
3. **Одна команда:**
   ```sh
   python tools/build_chapter.py <папка_с_аудио>
   ```
   она по очереди запускает:
   * **`align_chapter.py --write`** — авто-тайминг по мора-весу (гуру = 2, лагху = 1)
     и онсетам аудио; записывает поле `timing` прямо в JSON шлоки;
   * **`render_chapter.js`** — рендер `feed_v1` (вертикальный MP4 9:16 @ 30 к/с) плюс
     субтитры `.srt`/`.vtt` → `dist/`;
   * **`post_kit.py`** — подписи RU/EN, хэштеги и UTM-CTA для каждой площадки →
     `drop/<глава>/<id>/`.
4. **Полезные флаги:** `--dry-run` (показать план и команды, ничего не запуская),
   `--only id1,id2`, `--format 9:16|1:1`, `--skip-align/--skip-render/--skip-postkit`.
5. **Контроль качества:** слоги с низкой уверенностью авто-тайминга подсвечиваются
   оранжевым в редакторе тайминга — поправьте их там при необходимости. Сводка
   готовности в конце скажет, что ещё блокирует публикацию (например, отсутствие аудио).
6. **Расписание и постинг:** `python tools/schedule_drops.py --config schedule.yaml` строит
   план постов (`drop/schedule_plan.json`) из файла каденции (`schedule.example.yaml` →
   копировать в `schedule.yaml`). Флаг `--live` запускает постинг через публикаторы по
   площадкам (Telegram, VK, Facebook, Instagram, WordPress). **Безопасность:** публикатор
   срабатывает **только** при `--live` **и** наличии учётных данных площадки в окружении —
   иначе он отдаёт `skip` и не делает сетевых вызовов. Учётные данные — в Приложении D.

## 3. Менеджер прав — клиренс перед публикацией
**Цель:** гарантировать, что ни одно видео не уходит в публикацию, пока не очищены
**и аудио, и перевод**. Это не формальность: и распев, и русский перевод — чужая
интеллектуальная собственность.

### Что и как очищено:
| Слой | Источник | Статус | Где зафиксировано |
| :--- | :--- | :--- | :--- |
| **EN перевод** | Теланг (1882) | общественное достояние (`public-domain`) | `translation.rights.en` |
| **RU перевод** | Семенцов (под защитой в РФ до конца 2056) | **лицензирован** письмом-разрешением наследницы (дочери) — `cleared` | `translation.rights.ru`, `permission_ref: SK-LIC-2026-002/…` |
| **Аудио** | Уша Санка, `SK-LIC-2026-001` | договор-черновик + записей пока нет → **единственный оставшийся барьер** | `audio.license` |

### Рабочий процесс:
1. **Поле прав:** в каждом `verses/data/*.json` блок `translation.rights` хранит
   правообладателя, источник, лицензию, `permission_ref` и `status`.
2. **Автоматическая блокировка:** `validate_library.py` и `post_kit.py` не дают
   опубликовать непрояснённое — публиковаться может только `public-domain`,
   `own-work` или `cleared`. Непрояснённый RU-перевод порождает `caption_ru.BLOCKED.txt`
   вместо подписи и снимает у шлоки флаг «готова к публикации».
3. **Договоры:** генерируются скриптами — `docs/legal/make_agreement.js` (аудио,
   SK-LIC-2026-001) и `docs/legal/make_sementsov_agreement.js` (перевод, SK-LIC-2026-002).
   Черновики `.docx` остаются с незаполненными полями (имя/адрес/дата/объём) до
   получения данных от правообладателя.

## 4. Куратор библиотеки — Teaching Pipeline и Evals
**Цель:** поддерживать качество библиотеки шлок и следить за эффективностью платформы.

### Рабочий процесс:
1. **Прием контента:** проверьте Pull Request с новыми JSON-файлами шлок.
2. **Автоматическая валидация** — запускается **Teaching Pipeline** (LangGraph):
   * **VerseCurator** проверяет соответствие схеме;
   * **ContentEnricher** (через Gemini Flash) заполняет недостающие переводы (RU/EN) и теги
     — машинные переводы помечаются в `provenance`;
   * **QualityGate** гарантирует, что название метра соответствует метрической структуре;
   * **StudentAnalyzer** анализирует историю SRS и рекомендует очередь обучения.
3. **Контроль показателей:** `python tools/cost_dashboard.py` (бюджет ≈ **$0.10 на шлоку**),
   `python tools/student_stats.py` (шлоки, оказавшиеся сложнее ожидаемого).
4. **Бенчмаркинг:** `python evals/judge.py` — «золотой набор» из 8 случаев в
   `evals/golden/`; убедитесь, что переводы LLM не ухудшились после изменений в конвейере.

# Часть II. Конечные пользователи

## 5. Зритель → студент — воронка из соцсетей
**Цель (для проекта):** бесплатное вертикальное видео приводит зрителя на платный курс.
Пейволла нет — видео и есть воронка.

### Путь зрителя:
1. **Видит клип `feed_v1`** в Telegram / VK / Facebook / Instagram (или как пост в WordPress): тёмный вертикальный
   кадр — ॐ, название и метр (хук) → деванагари с построчной подсветкой слогов под
   распев (карaoке-заливка) → эндкард с переводом и кнопкой в финальные секунды.
2. **Нажимает CTA** `samskrtam.ru/usha-sanka` — ссылка размечена UTM-метками по площадке
   и шлоке, поэтому видно, **какие шлоки и шаблоны конвертируют**.
3. **Становится студентом курса** и/или переходит в браузерный плеер ниже.

## 6. Создатель контента (Учитель / Исследователь)
**Цель:** превратить сырой санскритский текст и аудио в структурированный урок для
библиотеки — он же станет исходником для конвейера видео.

### Рабочий процесс:
1. **Ввод текста:** откройте `index.html`, вставьте половины шлоки (s1/s2). Метр
   определится автоматически, построится волновая диаграмма.
2. **Интеграция аудио:** загрузите `.mp3`/`.wav`.
3. **Корректировка метрики:** правый клик по слогам исправляет вес (Гуру/Лагху), если
   автоматика пропустила нюанс (например, *muta cum liquida*).
4. **Синхронизация тайминга:** **Редактор тайминга** — сначала *Режим пад* (8 границ),
   затем *Режим слогов* (точная настройка горячими клавишами `Ctrl+стрелки`).
5. **Метаданные и экспорт:** заполните форму **Экспорт в библиотеку** (переводы,
   сложность, теги), скачайте `[id].json` в `verses/data/`. Экспортируйте **PNG** для
   печати и **Karaoke MP4** для соцсетей.
6. **Облачное сохранение:** сохраните сессию на Google Drive для будущих правок.

> Шлоки из библиотеки затем рендерятся пакетно в шаблоне `feed_v1` (см. сценарий 2),
> поэтому аккуратные метаданные и тайминг здесь экономят работу продюсеру дропа.

## 7. Активный студент (Ученик)
**Цель:** системно выучить шлоку и освоить её метрический ритм через интервальные повторения.

### Рабочий процесс:
1. **Поиск:** `catalogue.html` (или Telegram Mini App), фильтр по сложности/метру.
2. **Учебная сессия:**
   * **Фаза А (Полный режим):** слушайте аудио, глядя на подписи слогов.
   * **Фаза Б (Режим точек):** скройте текст, следуйте за пульсацией метра, напевая вслух.
   * **Фаза В (Вслепую):** только движущийся маркер — воспроизведите по памяти.
3. **Самопроверка:** **Квизы** (определить метр, вписать скрытый слог, отстучать ритм),
   затем оцените запоминание (😊/😐/😕) — алгоритм SM-2 спланирует повторения.
4. **Облачная синхронизация:** **Cloud Sync** сохраняет прогресс в Firebase между устройствами.

---

## 8. Оффлайн-практик (Мобильный пользователь)
**Цель:** продолжать обучение в дороге или при плохой связи.

### Рабочий процесс:
1. **Подготовка:** откройте шлоку один раз онлайн — данные сессии кэшируются.
2. **Оффлайн:** откройте `student.html?id=[id]`; **Service Worker** отдаёт интерфейс,
   **Drive Fallback** — данные шлоки из локального кэша.
3. **Индикатор:** в заголовке появляется **«Оффлайн-режим (кэш)»**.
4. **Синхронизация:** при восстановлении связи `progress.html` синхронизирует результаты
   с Firebase.

# Приложение A — Чек-лист первого дропа (Продюсер дропа)

От аудио главы до запланированных постов:

1. ☐ **Шлоки заведены** в `verses/data/` для каждой записанной строки; `python tools/validate_library.py` проходит.
2. ☐ **Права очищены** — `validate_library.py` не выдаёт предупреждений *«not cleared to publish»* (EN/RU), лицензия на аудио на месте.
3. ☐ **Аудио названо** `<verse_id>.<ext>` (напр. `bhg_2_47.m4a`), всё в одной папке.
4. ☐ **Puppeteer установлен** — `npm install --prefix tools` (один раз).
5. ☐ **Сухой прогон** — `python tools/build_chapter.py <папка> --dry-run` показывает верное сопоставление шлок и отсутствие «осиротевшего» аудио.
6. ☐ **Сборка** — `python tools/build_chapter.py <папка>` → `dist/*.mp4` + `.srt`/`.vtt` + `.png`, и `drop/<глава>/<id>/`.
7. ☐ **QA тайминга** — проверьте оранжевые (неуверенные) слоги в редакторе тайминга; пере-выровняйте или поправьте вручную.
8. ☐ **Готовность** — в `manifest.json` каждой шлоки `ready_to_publish: true` (без барьеров).
9. ☐ **Расписание** — скопируйте `schedule.example.yaml` → `schedule.yaml`, затем `python tools/schedule_drops.py` → `drop/schedule_plan.json`.
10. ☐ **Публикация** — `python tools/schedule_drops.py --live` постит на каждую площадку, у которой заданы учётные данные (для Telegram достаточно токена от @BotFather); остальные аккуратно пропускаются. Учётные данные — в Приложении D.

# Приложение C — Глоссарий

- **гуру / лагху** — тяжёлый / лёгкий слог; метрические веса, которые показывает волновая диаграмма (гуру — тёмно-красный, лагху — тёмно-зелёный).
- **пада** — четверть шлоки; в ануштубхе 4 пады по 8 слогов.
- **мора** — единица просодического тайминга; выравниватель считает гуру = 2 моры, лагху = 1.
- **feed_v1** — вертикальный шаблон (1080 × 1920) для соцсетей (`src/core/feed.js`).
- **дроп-кит** — папка `drop/<глава>/<id>/`: подписи, хэштеги, UTM-CTA и `manifest.json`.
- **барьер публикации** — причина, по которой шлоку пока нельзя публиковать (непрояснённые права, нет аудио); проверяется `validate_library.py` и `post_kit.py`.
- **UTM** — метки кампании в CTA-ссылке, позволяющие воронке измерять, какие шлоки и шаблоны конвертируют.
- **клиренс / `cleared`** — источник перевода или аудио, подтверждённый как законный для публикации (`public-domain`, `own-work` или лицензирован).

# Приложение D — Учётные данные публикаторов

`python tools/schedule_drops.py --live` постит через `tools/publishers.py`. Публикатор
срабатывает **только** при `--live` **и** наличии всех его переменных окружения — иначе он
пропускается без сетевых вызовов. Храните это в `.env` (в .gitignore) — никогда не коммитьте.

| Площадка | Переменные окружения | Где взять | Примечания |
| :--- | :--- | :--- | :--- |
| **Telegram** | `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHANNEL_ID` | @BotFather → `/newbot`; добавить бота админом канала | Без проверки — проще всего, начните отсюда. Заливает локальный MP4. |
| **VK** | `VK_ACCESS_TOKEN`, `VK_OWNER_ID` | токен приложения/сообщества VK; `VK_OWNER_ID` = id стены сообщества (отрицательный) | Загрузка через `video.save` → `wall.post`. |
| **Facebook** | `FB_PAGE_ID`, `FB_PAGE_ACCESS_TOKEN` | developers.facebook.com → приложение → токен доступа Страницы | Постит в видео Страницы. |
| **Instagram** | `IG_BUSINESS_ACCOUNT_ID`, `IG_ACCESS_TOKEN`, `IG_VIDEO_BASE_URL` | Meta-приложение + IG Business/Creator; App Review для `instagram_content_publish` | **Нужен MP4 по публичному URL** (`IG_VIDEO_BASE_URL`/`<id>_9x16.mp4`) — IG не принимает локальный файл. |
| **WordPress** | `WP_BASE_URL`, `WP_USER`, `WP_APP_PASSWORD` | WP Admin → Пользователи → Пароли приложений | Заливает видео и создаёт **черновик** записи. |

### Быстрый старт с `.env`

Скопируйте **`.env.example`** → **`.env`** (в .gitignore), заполните нужные площадки, затем:

```sh
python tools/schedule_drops.py --live
```

`--live` автоматически загружает `.env`. Площадка постит только когда заданы **все** её
переменные; остальные пропускаются без сетевых вызовов. Без `--live` (или без учётных данных)
каждая запись показывается со статусом `skip` / `dry-run`, ничего не отправляется. Telegram,
VK, Facebook и WordPress заливают локальный MP4 из `dist/` напрямую; Instagram — исключение:
он забирает файл по публичному URL (`IG_VIDEO_BASE_URL`).

### Где взять каждый

**Telegram** — без проверки, начните отсюда.
1. Напишите **@BotFather** → `/newbot` → скопируйте **токен** бота → `TELEGRAM_BOT_TOKEN`.
2. Создайте канал и добавьте бота **админом** с правом публикации.
3. `TELEGRAM_CHANNEL_ID` = `@публичноеимя` или числовой `-100…` id приватного канала (перешлите пост канала боту `@userinfobot`, чтобы узнать id).

**VK**
1. Создайте **Standalone**-приложение на [dev.vk.com](https://dev.vk.com/).
2. Получите токен со scope `video` + `wall` — токен сообщества (Сообщество → Управление → Работа с API → Создать ключ) или OAuth-токен со `scope=video,wall,offline` → `VK_ACCESS_TOKEN`.
3. `VK_OWNER_ID` = id стены сообщества **отрицательным** числом (напр. `-123456789`); числовой id — через [regvk.com/id](https://regvk.com/id/) или исходник страницы группы.

**Facebook** (Страница)
1. [developers.facebook.com](https://developers.facebook.com/) → **Create App** (Business).
2. В Graph API Explorer выберите Страницу и выдайте `pages_manage_posts`, `pages_read_engagement`, `pages_show_list`.
3. Обменяйте на **долгоживущий токен Страницы** (Access Token Debugger → «Extend Access Token») → `FB_PAGE_ACCESS_TOKEN`.
4. `FB_PAGE_ID` = Страница → О странице → «Прозрачность Страницы» (или `GET /me/accounts`).

**Instagram** (Reels — нужен публичный URL видео)
1. Переведите аккаунт IG в **Business/Creator** и привяжите к Странице Facebook.
2. То же Meta-приложение; запросите `instagram_basic` + `instagram_content_publish` (**App Review** + верификация бизнеса для боевого режима) → `IG_ACCESS_TOKEN`.
3. `IG_BUSINESS_ACCOUNT_ID` = `GET /{page-id}?fields=instagram_business_account`.
4. `IG_VIDEO_BASE_URL` = публичный базовый URL, отдающий `dist/<id>_9x16.mp4` (напр. `https://samskrtam.ru/karaoke/video`).

**WordPress** (создаёт черновик)
1. `WP_BASE_URL` = корень сайта, напр. `https://samskrtam.ru` (без слэша в конце).
2. WP Admin → **Пользователи → Профиль → Пароли приложений** → задайте имя → «Добавить новый пароль приложения» → скопируйте → `WP_APP_PASSWORD` (WordPress 5.6+ по HTTPS).
3. `WP_USER` = ваш логин WordPress.
